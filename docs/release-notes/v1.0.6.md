## 架构改进 / Architecture Improvements

### 方案 A：编辑器侧等待探针就绪信号（已实现 ✅）

| 项 | 说明 / Description |
|---|---|
| 文件 / File | `mcp_debugger_bridge.gd` |
| 原理 / Principle | 探针注册 `EngineDebugger.register_message_capture()` 成功后发送 `mcp:probe_ready` 信号。Bridge 在 `_capture()` 中识别此信号并标记 session 就绪，`_request_runtime_probe_poll()` 在发请求前先 `await bridge.wait_for_probe_ready()` 等待就绪 |
| 效果 / Effect | 消除"请求在探针就绪前发出"的竞态窗口，轻量级请求首次即成功 |
| 改动 / Changes | 新增 `_probe_ready_session_ids`、`is_probe_ready()`、`wait_for_probe_ready()`、`reset_probe_ready()` |

### 方案 D：MCP 工具执行全量异步化（已实现 ✅）

| 项 | 说明 / Description |
|---|---|
| 文件 / File | `mcp_server_core.gd` + `debug_tools_native.gd` + `mcp_debugger_bridge.gd` |
| 原理 / Principle | `OS.delay_msec()` 阻塞编辑器主线程 → `EditorDebuggerPlugin._capture()` 无法调度 → 游戏回复到达 IPC 通道但编辑器无法处理。改用 `await SceneTree.process_frame` 让主循环推进，`_capture()` 正常调度 |
| 效果 / Effect | 重量级请求（截图、场景树）首次调用即成功，无需重试 |
| 改动 / Changes | 信号连接 lambda 启动协程；`_handle_request` / `_handle_tool_call` / `_handle_resource_read` 改为 async；`_request_runtime_probe_poll` 和 `wait_for_probe_ready` 中 `OS.delay_msec` → `await tree.process_frame`；35 处 `_request_runtime_probe_poll()` 调用补加 `await` |
| 改动量 / LOC | ~51 行核心改动 |

### 方案 A + D 联合效果

| 请求类型 / Request Type | 修复前 / Before | 修复后 / After |
|---|---|---|
| 轻量级（get_runtime_info、performance_snapshot） | stale 回退 / 首次 pending | ✅ 首次即成功 |
| 重量级（get_runtime_screenshot、get_runtime_scene_tree） | 多次 pending | ✅ 首次即成功 |
